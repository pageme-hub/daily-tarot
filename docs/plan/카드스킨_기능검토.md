## 1. MVP 포함 여부 판단: v1.1로 분리 권장

### 판단 근거

**MVP에서 제외하고 v1.1로 분리해야 하는 이유:**

| 기준 | 분석 |
|------|------|
| RULES.md MVP 원칙 | "핵심 기능 3개 이하로 제한" -- 현재 MVP 3개 슬롯이 이미 꽉 참 (오늘의 카드 / 도감 / 저장-공유) |
| 3주 출시 원칙 | 인앱결제는 스토어 심사, 결제 테스트, 환불 정책 등 부가 작업이 크다. 2주 목표에 최소 1~1.5주 추가 |
| 수익 구조 충돌 | 현 MVP는 광고 수익 모델. 인앱결제를 추가하면 광고+IAP 혼합 모델이 되어 초기 UX 설계 복잡도 증가 |
| 검증 순서 | 먼저 DAU/리텐션으로 "사용자가 매일 돌아오는지"를 검증한 후, 스킨 결제 의향을 측정하는 것이 합리적 |
| 이미지 에셋 준비 | 78장 x N세트 = 수백 장의 이미지 생성 + 품질 검수 시간이 필요 |
| 인앱결제 심사 | Google Play 인앱결제 첫 설정 + 심사 리드타임이 MVP 출시를 지연시킬 수 있음 |

**단, 아키텍처에 스킨 확장을 미리 반영해야 한다:**

현 MVP의 `image_url` 필드와 Storage 구조를 스킨 확장을 고려한 형태로 설계해두면, v1.1에서 코드 변경을 최소화할 수 있다. 이것은 "구현하지 않되, 대비는 해두는" 전략이다.

### v1.1 출시 시점 기준

- DAU 100+ 안정 달성
- 1주일 리텐션 20% 이상
- 스토어 리뷰에서 "디자인 좋다 / 다른 스타일 원한다" 류의 피드백 확인

### 2-2. 데이터 모델

#### Supabase 테이블 추가

```sql
-- 스킨 마스터
CREATE TABLE card_skins (
  id TEXT PRIMARY KEY,              -- "minimal_pastel", "classic", "dark_mystic" 등
  name TEXT NOT NULL,               -- "미니멀 파스텔"
  name_kr TEXT NOT NULL,            -- 한글명
  description TEXT,                 -- 스킨 설명
  price_tier TEXT NOT NULL,         -- "free" / "tier_1" / "tier_2"
  price_krw INT DEFAULT 0,         -- 표시용 가격 (원)
  store_product_id TEXT,            -- Google Play 인앱결제 상품 ID
  preview_image_urls TEXT[],        -- 미리보기용 대표 이미지 URL 6개
  is_active BOOLEAN DEFAULT true,   -- 활성/비활성 (서버 제어)
  sort_order INT DEFAULT 0,
  app_id TEXT NOT NULL DEFAULT 'daily_tarot',
  created_at TIMESTAMPTZ DEFAULT now()
);

-- 스킨별 카드 이미지 매핑
CREATE TABLE card_skin_images (
  id SERIAL PRIMARY KEY,
  skin_id TEXT NOT NULL REFERENCES card_skins(id),
  card_id INT NOT NULL REFERENCES tarot_cards(id),
  image_url TEXT NOT NULL,           -- Supabase Storage 경로
  thumbnail_url TEXT,                -- 썸네일 URL (선택)
  app_id TEXT NOT NULL DEFAULT 'daily_tarot',
  UNIQUE(skin_id, card_id)
);

CREATE INDEX idx_card_skin_images_skin ON card_skin_images(skin_id);

ALTER TABLE card_skins ENABLE ROW LEVEL SECURITY;
ALTER TABLE card_skin_images ENABLE ROW LEVEL SECURITY;

CREATE POLICY "card_skins_read" ON card_skins FOR SELECT USING (true);
CREATE POLICY "card_skin_images_read" ON card_skin_images FOR SELECT USING (true);
```

## 3. 기술적 구현 고려사항

### 3-1. Supabase Storage 이미지 저장 구조

```
card-images/
  daily_tarot/
    minimal_pastel/          <-- 기본 스킨 (무료)
      1.png                  <-- card_id 기준
      2.png
      ...
      78.png
      back.png               <-- 카드 뒷면
      thumbnails/
        1_thumb.png
        ...
    classic/                 <-- 유료 스킨
      1.png
      ...
      78.png
      back.png
      thumbnails/
        ...
    dark_mystic/
      ...
```

**MVP 단계에서의 사전 준비:**
- 현재 `card-images/daily_tarot/{card_id}.png` 구조를
- `card-images/daily_tarot/minimal_pastel/{card_id}.png`로 변경
- `tarot_cards.image_url`을 스킨 독립적인 기본 URL로 유지하되, 실제 이미지 로드 시 `{base_url}/{active_skin_id}/{card_id}.png` 패턴으로 조합하는 헬퍼 함수 준비

### 3-2. Hive 구매/선택 상태 캐싱

| Box | 용도 | 키 |
|-----|------|-----|
| `skin_purchase_box` | 구매 완료된 스킨 목록 | skinId -> SkinPurchaseCache |
| `active_skin_box` | 현재 적용 중인 스킨 | "active" -> ActiveSkinCache |
| `skin_download_box` | 다운로드 완료된 스킨 | skinId -> bool |

**오프라인 대응:**
- 구매 상태는 Hive에 캐싱하되, 앱 실행 시 RevenueCat과 동기화
- RevenueCat의 `customerInfo`가 source of truth, Hive는 오프라인 캐시

### 3-3. 이미지 프리로드/다운로드 전략

78장 x N스킨은 이미지 양이 많으므로 전략이 중요하다.

**권장 전략: "구매 후 일괄 다운로드"**

```
[구매 완료]
    |
    v
[다운로드 확인 다이얼로그]
"78장의 카드 이미지를 다운로드합니다 (약 15MB). Wi-Fi 환경을 권장합니다."
    |
    v
[백그라운드 다운로드 + 프로그레스 바]
    |
    +-- 전체 완료 --> "적용 가능" 상태로 전환
    +-- 중단/실패 --> "이어받기" 지원 (다운로드된 파일 수 추적)
```

**이미지 사이즈 최적화:**
- 원본: 512x768 또는 768x1152 (카드 비율 2:3)
- 썸네일: 128x192 (도감 그리드용)
- WebP 형식 권장 (PNG 대비 30~50% 용량 절감)
- 78장 x WebP = 약 10~20MB per skin

**캐싱 계층:**

```
1. 메모리 캐시 (cached_network_image 기본)
2. 디스크 캐시 (cached_network_image 기본)
3. Supabase Storage (원본)
```

**기본 스킨 처리:**
- 기본 스킨(minimal_pastel)은 앱 에셋에 번들링 (assets/images/cards/)
- 네트워크 없이도 기본 스킨은 항상 사용 가능
- 유료 스킨만 다운로드 방식

### 3-4. 인앱결제 패키지 비교

| 기준 | revenue_cat (purchases_flutter) | in_app_purchase |
|------|-------------------------------|-----------------|
| 복잡도 | 낮음 (서버사이드 영수증 검증 자동) | 높음 (직접 영수증 검증 필요) |
| 서버 비용 | 월 매출 $2,500 이하 무료 | 무료 |
| 구매 복원 | 자동 지원 | 직접 구현 |
| 크로스 플랫폼 | iOS/Android 통합 관리 | 각 플랫폼 별도 |
| 분석 대시보드 | 제공 | 없음 (직접 구축) |
| 의존성 | RevenueCat 서비스 의존 | Google/Apple 직접 연동 |

**권장: RevenueCat (purchases_flutter)**

이유:
1. 매일타로의 예상 매출 규모에서 무료 티어로 충분
2. 영수증 검증, 구매 복원, 구독 관리를 서버 없이 처리
3. 파생 앱(매일오라클, 매일룬)에서 동일 패턴 재사용
4. 1인 개발/소규모 팀에서 인앱결제 직접 구현은 리스크가 큼

**상품 구조 (비소모품):**

| 상품 ID | 이름 | 가격 (KRW) |
|---------|------|-----------|
| `skin_classic` | 클래식 올드스쿨 | 3,900 |
| `skin_asian` | 아시안 동양풍 | 3,900 |
| `skin_dark_mystic` | 다크 미스틱 | 3,900 |
| `skin_watercolor` | 수채화 | 3,900 |
| `skin_bundle_all` | 전체 스킨 팩 | 9,900 |

비소모품(Non-consumable)으로 설정하여 한 번 구매하면 영구 소유.

## 4. 파생 앱과의 스킨 공유 가능성

### 공유 가능한 부분

| 항목 | 공유 가능 여부 | 설명 |
|------|-------------|------|
| 스킨 시스템 코드 | O (높음) | `SkinProvider`, `SkinRepository`, `SkinShopScreen` 등 로직 재사용 |
| RevenueCat 연동 코드 | O (높음) | 결제 로직은 거의 동일 |
| 다운로드 관리자 | O (높음) | 이미지 다운로드 전략 동일 |
| 스킨 이미지 자체 | X (불가) | 카드 수가 다름 (타로 78장, 오라클 44장, 룬 24개) |
| 스킨 스타일/컨셉 | 부분적 | 같은 프롬프트 스타일을 적용할 수 있으나 카드 특성이 달라 완전 동일 불가 |
| Supabase 테이블 구조 | O (높음) | `card_skins`, `card_skin_images` 테이블을 `app_id`로 구분하여 공유 |
| 가격 정책 | O (높음) | 동일 가격 체계 적용 가능 |

### 공유 아키텍처 권장

```
app-factory/
  packages/
    skin_module/             <-- 공유 패키지로 추출
      lib/
        data/
          models/skin.dart
          repositories/skin_repository.dart
        providers/
          skin_provider.dart
          purchase_provider.dart
        ui/
          screens/skin_shop_screen.dart
          widgets/skin_preview_card.dart
        utils/
          image_download_manager.dart
```

매일타로에서 v1.1 완성 후, 스킨 시스템을 `skin_module` 패키지로 추출하면 매일오라클/매일룬에서 의존성 추가만으로 재사용 가능하다.

### 크로스 앱 번들 가능성

"매일 시리즈 올인원 스킨 팩" 같은 크로스 앱 번들은 Google Play 정책상 앱 간 구매 공유가 어렵다. 대신:
- 각 앱에서 별도 인앱결제로 판매
- "매일타로에서 클래식 스킨 구매 시, 매일오라클에서 할인 코드 제공" 같은 프로모션은 RevenueCat의 프로모셔널 오퍼로 가능

## 5. 리스크 및 주의사항

### 높음

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 이미지 용량 폭증 | 유료 스킨 4개 x 78장 = 312장 추가. 사용자 스토리지 부담 | WebP 변환 필수, 스킨당 15MB 이하 목표, 미사용 스킨 캐시 삭제 옵션 |
| DALL-E 스타일 일관성 | 78장을 동일 스타일로 일관되게 생성하기 어려움. 일부 카드만 이질적이면 품질 저하 | 프롬프트에 스타일 앵커 강화, 생성 후 수동 검수 필수, 품질 미달 카드 재생성 |
| 인앱결제 심사 지연 | Google Play 인앱결제 첫 설정 시 심사 추가 소요 (1~3일) | v1.0 출시 후 바로 결제 설정 준비 시작 |
| 환불 처리 | 디지털 콘텐츠 환불 정책 미비 시 스토어 경고 | Google Play 환불 정책 자동 적용 (48시간 이내), 앱 내 환불 안내 문구 |

### 중간

| 리스크 | 영향 | 대응 |
|--------|------|------|
| 결제 전환율 저조 | 무료 사용자가 유료 스킨 구매를 안 함 | 미리보기 충분히 제공, 기간 한정 할인, "오늘의 카드"에서 다른 스킨 카드 맛보기 노출 |
| 광고 수익 잠식 | 인앱결제 사용자가 광고 제거를 기대 | 스킨 구매와 광고 제거는 별개임을 명확히. 광고 제거는 별도 상품으로 분리 |
| 기본 스킨 매력도 하락 | 유료 스킨 강조하다가 기본 스킨이 "싸구려"로 인식됨 | 기본 스킨도 충분히 고품질로 유지. "미니멀 파스텔"이라는 명확한 아이덴티티 부여 |

### 낮음

| 리스크 | 영향 | 대응 |
|--------|------|------|
| Supabase Storage 비용 | 이미지 수 증가로 스토리지 + CDN 비용 증가 | 무료 티어 1GB + 2GB 전송으로 초기 충분. 스킨 4세트 기준 약 100MB |
| RevenueCat 서비스 장애 | 결제 불가 상태 발생 | 구매 상태 Hive 캐시로 기존 구매 스킨은 오프라인 사용 가능 |

## 6. MVP 사전 준비 체크리스트 (v1.0에서 해둘 것)

v1.1에서 스킨 기능을 추가할 때 최소한의 변경으로 확장하려면, v1.0에서 아래를 미리 반영해둔다.

- [ ] Storage 경로를 `daily_tarot/minimal_pastel/{card_id}.png` 형태로 구성
- [ ] 이미지 URL 조합을 헬퍼 함수로 추상화 (`getCardImageUrl(cardId, skinId)`)
- [ ] Hive typeId 3, 4를 스킨용으로 예약 (다른 모델에 사용하지 않기)
- [ ] `tarot_cards` 테이블의 `image_url` 컬럼은 기본 스킨 URL로 유지하되, 스킨 전환 시 무시 가능하도록 설계
